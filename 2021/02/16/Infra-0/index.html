제1장. 인프라 아키텍처

작성일 2021-02-16 Edited on 2021-03-01 In IT infrastructure

글을 시작하기에 앞서 해당 글은 ‘그림으로 공부하는 IT 인프라 구조’를 바탕으로 요약 및 정리한 글임을 알려드립니다.

시작하기에 앞서

간단한 웹 대시보드를 사이드 프로젝트로 진행하다보니 여러 가지 의문점들이 생겼다. 웹서버가 정확히 어떤 형태로 돌아가는 건지, DB와 웹 어플리케이션은 어떻게 연결해주면 되는지 등등 말이다. 위 질문의 대답들은 대부분 구글링을 통해서 해결했지만, 왜 그렇게 해야 하는지에 대한 설명은 없었다. 또 누군가 설명해준 것을 봐도 깊이 이해하기가 어려웠다. 이런 케이스들을 묶어서 살펴보니 결국 내가 컴퓨터를 잘 모르기 때문에 발생했다는 것을 깨달았다. 컴퓨터가 어떻게 작동하는지, 또 하드웨어에 따라 소프트웨어 구조가 어떻게 결정되는지 등의 대한 것들을 공부해보고 싶었다. 그래서 ‘그림으로 공부하는 IT 인프라 구조’라는 책으로 공부하며, 블로그에 내용을 정리하고자 한다.

인프라란 무엇인가?

IT 인프라는 IT의 기반이 되는 것들을 통칭하는 개념이다. 핸드폰이 작동하는 구조나, 은행 앱이 작동하는 구조 등의 특정 구조들이 IT 인프라에 속한다. 그렇다면 인프라 아키텍처는 무엇인가? 위 예시에서 가져온다면 핸드폰이 작동하는 구조나 은행 앱이 작동하는 구조일 것이다.

궁극의 아키텍처는 없다

만능 아키텍처 하나로 모든 프로젝트에 적용할 수 있을까? 위 제목에서 볼 수 있듯이 아니다. 아키텍처 설계 요소에는 반드시 장점과 단점이 공존한다. 상황에 띠라서 시스템의 가장 중요한 장점은 살리고 단점은 최소화하도록 설계하는 것이 중요하다.

아키텍처

IT 인프라는 컴퓨터로 구성된다. 기본적인 구성방식에는 집약형과 분할형이 있다.

집약형 아키텍처

집약형 아키텍처는 대형 컴퓨터 한 대로 모든 업무를 처리하는 구조이다. 장점으로는 컴퓨터 한 대만 있으면 되므로 구성이 간단하고, 리소스 관리나 이중화 덕분에 안정성이 높고 고성능이라는 점이 있다.

  • 리소스 관리: 복수의 다른 업무처리를 동시에 실행할 수 있도록 하여 한 업무에서 고부하 처리 요구가 와도 다른 처리에 영향을 주지 않게 하는 것이다.
  • 이중화: 컴퓨터를 구성하는 주요 부품들을 다중화하여 하나 고장나도 업무가 지속될 수 있게 하는 것이다.

단점으로는 대형 컴퓨터의 도입 및 유지 비용이 비싸고, 확장성의 한계 등이 있다.

분할형 아키텍처

분할형 아키텍처는 여러 대의 컴퓨터를 조합하여 하나의 시스템을 구축하는 구조이다. 이 아키텍처는 표준 OS나 개발 언어를 사용하기 때문에 오픈 시스템이라고도 불리며, 여러 대의 컴퓨터를 연결한다는 점에서 분산 시스템이라고도 불린다. 장점으로는 저렴한 비용과 서버를 쉽게 늘릴 수 있어 확장성이 높다는 점이 있으며, 단점으로는 대수가 늘어날수록 관리가 어렵고 한대가 망가질 시 피해의 최소화를 위한 구조를 검토해야 한다는 점이다.

  • 서버: 서버라는 단어는 하드웨어, 소프트웨어에서 전부 사용된다. 분산처리에 사용되는 물리적인 컴퓨터 등을 물리 서버라고 하며, 웹서버나 DB 서버와 같이 소프트웨어로 구현된 서버를 논리 서버라고 부른다.

분할형에는 서버마다 다른 작업을 할당하는 수직 분할형 아키텍처와 비슷한 작업을 여러 서버에 할당하는 방법인 수평 분할형 아키텍처가 있다.

수직 분할형 아키텍처

위에서 설명했듯 수직 분할형 아키텍처는 각 서버마다 역할을 분리하는 것이다. 수직형이라고 표현한 이유는 역할에 따라 서버의 계층이 위 또는 아래로 나눠지기 때문이다. 수직 분할형 아키텍처는 크게 클라이언트-서버형 아키텍처, 3계층형 아키텍처으로 나뉜다.

클라이언트-서버형 아키텍처

클라이언트-서버형은 클라이언트 측에 전용 소프트웨어를 설치해서 역할을 분리하는 것이다. 노트북, 핸드폰, 태블릿 등 클라이언트 측의 소형 컴퓨터는 화면 포시나 단순한 계산을 실행하고, 물리 서버는 데이터베이스나 미들웨어 등을 바탕으로 클라이언트의 요청을 처리한다. 해당 아키텍처는 클라이언트 측에서 일정 부분 해결해주기 때문에 소수의 서버가 다수의 클라이언트를 처리할 수 있다는 장점이 있다. 하지만 물리 서버의 일부가 변경될 때 마다 클라이언트 소프트웨어도 지속적으로 업데이트 해야 하고, 사용자가 업데이트를 해준다는 보장도 없기에 보안이 위험해질 수도 있다.

3계층형 아키텍처

3계층형은 클라이언트-서버형을 발전시킨 형태이다. 여기서 3계층은 ‘프레젠테이션 계층’, ‘어플리케이션 계층’, ‘데이터 계층’을 의미한다. 각 계층들의 이름만으로도 알 수 있듯이 클라이언트-서버형보다 역할이 명확하게 구분되어 있다.

  • 프레젠테이션 계층: 사용자 입력 수집, 웹 브라우져 화면 표시
  • 어플리케이션 계층: 사용자 요청(request)에 따라 업무처리
  • 데이터 계층: 어플리케이션의 요청에 따라 데이터 입출력

시스템 순서는 다음과 같다. 먼저 사용자는 웹 브라우저를 통해 시스템에 접속하며, 사용자 요청은 프레젠테이션 계틍에 있는 웹 서버에 전달된다. 웹 서버는 그 요청을 어플리케이션 계층에 있는 서버(AP서버)에 전달하고, AP 서버는 해당 요청을 판단하여 필요한 데이터들을 데이터 계층에 있는 DB에 요청한다.
이런 아키텍처는 클라이언트-서버형에 비해 각 서버들의 부하가 상대적으로 적고, 클라이언트 단말의 부가적인 업데이트도 필요 없다. 다만, 규모가 큰 시스템을 해당 구조로 운영하면 클라이언트-서버형보다는 복잡해진다.

수평 분할형 아키텍처

수직 분할형 아키텍처에서 시스템을 더 확장하기 위해서는 각각의 역할을 담담하는 서버들의 숫자를 증대해야 한다. 이와 샅이 하나의 역할에 여러 서버를 사용하는 구조를 수평 분할형 아키텍처라고 한다. 하나의 역할을 여러 서버가 나눠서 하게 된다면, 서버 한대가 문제가 생겨도 시스템 전반에 큰 타격을 주지 않기에 안전성도 높다. 대부분의 시스템이 수직 분할형과 수평 분할형을 모두 사용하고 있다.

단순 수평 분할형 아키텍처

단순 수평 분할형 아키텍처는 기준점에 따라 시스템을 완전하게 분할하는 구조이다. 예를 들어 기준점이 지역이라고 가정해보자. 그러면 한 프로그램의 시스템은 서울지사와 부산지사로 나눠져 있고, 이 시스템들이 완전 독립적으로 작동을 하는 것이다. 따라서 서울지사에 있는 시스템에 문제가 생겨도 부산지사 시스템은 잘 작동된다. 이런 독립성은 안정성을 향상시켜주기에 장점이라고 할 수 있지만 동시에 단점도 발생한다. 각 시스템들이 완전히 독립적이기에 데이터 또한 각자 관리되며, 사용되는 어플리케이션도 각각 업데이트를 진행해야 한다. 또한 사용자 수에 따라서 한쪽의 시스템만 과부하될 수도 있다.

공유형 아키텍처

공유형 아키텍처는 데이터 계층을 동기화 한 구조이다. 위의 예시를 본다면 서울지사 시스템의 데이터 계층과 부산지사 시스템의 데이터 계층이 상호작용할 수 있도록 설계하는 것이다. 해당 구조는 서울지사에서도 부산 데이터를 확인할 수 있고, 부산지사에서도 서울 데이터를 가져갈 수 있다.

지리 분할형 아키텍처

지리 분할형 아키텍처는 시스쳄 자체의 효율성보다는 연속성과 가용성을 높이기 위한 구조이다. 즉 시스템 일부가 고장나거나 네트워크 상의 문제가 생겨도 계속 작동할 수 있는 구조라고 이해하면 된다.

스탠바이형 아키텍처

스탠바이형 아키텍처는 물리 서버 최소 2대를 준비하여 1대가 고장났을 시 가동중인 소프트웨어를 다른 1대로 옮겨 운영하는 구조이다. 이때 소프트웨어 재시작을 자동으로 하는 구조를 페일오버(Failover)라고 한다. 해당 방식은 작동하는 물리 서버 고장에 대처할 수 있지만, 평소에는 페일오버 서버가 놀고 있기 때문에 리소스 낭비가 발생한다. 따라서 서버 모두를 교차로 사용하다가 한 대가 문제생기면 나머지 서버가 모든 일을 하는 방식으로 많이 사용된다.

재해대책형 아키텍처

최근 들어 재해 발생이 현실적인 문제가 되고 있다. 재해가 발생하여 해당 지역의 서버들을 사용할 수 없을 때를 대비하는 구조가 재해대책형 아키텍쳐이다. 재해는 지역별로 발생하는 경우가 많기에 보통은 지역을 다르게 설정하여 재해대책형 사이트를 만든다. 또한 기존에 사용하는 서버들의 데이터를 동기화하여 문제가 생겼을 시 바로 재해대책형 사이트로 연결될 수 있도록 구성한다. 따라서 해당 구조에서 가장 중요한 부분은 재해대책용 시스템의 어플리케이션 최신화, 데이터 동기화이다.
